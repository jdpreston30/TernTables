### 1 — Methods document overhaul: per-table accuracy, ternB consolidation, bug fixes (v1.6.3.9XXX)

**What changed in the package:**

#### Single-table runs (`ternD` / `ternG` calling `write_methods_doc` directly)

- Previously, `write_methods_doc()` always wrote **three boilerplate paragraphs** regardless of what was actually run — one for `ternD`, one for `ternG` two-group, one for `ternG` three-or-more-group — with generic section headers like *"If you used the package to generate two-group comparison tables (ternG)."*
- The function now writes **exactly one paragraph** corresponding to the specific configuration that was actually run, preceded by a bold **"Statistical Methods"** header and followed by a small italic attribution footer:
  > *"Generated by TernTables vX.X.X. This paragraph reflects the specific analysis configuration used in this run. For documentation of other configurations, see the package vignette."*
- The paragraph is fully dynamic — detects which tests actually ran from the output table, branches on `OR_col`, `OR_method`, `post_hoc`, and `source`, and generates only sentences relevant to what happened. No text about tests not used is included.

#### `ternB` combined-document runs (`ternB(methods_doc = TRUE)`)

- Previously, all tables were pooled and **one single paragraph** was generated using the most complex configuration across the entire bundle — a bundle of `ternD` + `ternG` two-group + `ternG` three-group would produce a single three-group paragraph that did not accurately describe the `ternD` table at all.
- Now replaced with a **per-table approach**: one labeled methods paragraph is generated for each table in the bundle, with the section heading taken from `table_caption` if set, or falling back to `"Table 1"`, `"Table 2"`, etc.
- If multiple tables in the bundle were run with **identical settings and produced identical paragraph text**, they are consolidated into a single section with their labels joined (e.g. *"Table 1 / Table 3"*) to avoid redundancy.
- Footer reads:
  > *"Each section reflects the specific configuration of the corresponding table(s). Sections with identical configurations are consolidated to avoid redundancy."*

#### Three metadata bugs fixed (affected `ternB` methods document accuracy)

1. `post_hoc` was not stored in the table metadata attached to each `ternG` output — `ternB(methods_doc = TRUE)` always generated *"pairwise post-hoc comparisons were not performed"* even when `post_hoc = TRUE` was used. **Fixed.**
2. `OR_method` was not stored in metadata — the Wald-only OR description was never reachable through `ternB`. **Fixed.**
3. The OR sentence was missing from three-or-more-group paragraphs — OR content was silently dropped when a `ternB` bundle contained a two-group OR table alongside a three-group table. **Fixed.**

#### New `boilerplate` parameter on `write_methods_doc()`

- `boilerplate = FALSE` (default) — normal behaviour, unchanged.
- When `TRUE`, ignores all other arguments and writes a comprehensive reference document covering all five standard TernTables configurations to `comprehensive_boilerplate_methods.docx` in the working directory.
- **This is a developer/reference feature only — never expose or trigger this in the app.** The app passes all defaults and this parameter will never be activated.

#### Odds ratio language

- All references to odds ratios across the README, vignette, and function documentation now consistently say **"unadjusted odds ratios"** to make clear no multivariable correction is applied.

**App changes needed:**

- **No changes to how the app calls `write_methods_doc()` or `ternB()` are needed** — function signatures are unchanged and all defaults are correct for the app's use case.
- **Critical verification:** when the app calls `ternB(methods_doc = TRUE)`, confirm it is passing the **tibble objects produced in the current session** (not reloaded from disk), since the per-table metadata (`ternB_meta` attribute) lives in memory on each returned tibble and is required for per-table paragraph generation. If the app already holds session tibbles in memory when calling `ternB`, no code changes are needed.
- **Visible change to users:** single-table methods documents now contain one clean paragraph under a bold header; combined `ternB` methods documents now contain one labeled section per unique table configuration, consolidated where configs are identical. Both are downloaded as a single methods `.docx` file as before.

### 3 — Feature: Multi-tab Excel preprocessor (app-side, no package change)

**What changed in the package:** N/A — this is a new app-side preprocessing feature.

**App changes needed:**

When a user uploads an Excel file, before any TernTables preprocessing runs, the app should check whether the file has **multiple tabs/sheets**:

#### Single sheet detected
- Import normally; no dialogue needed. Proceed with standard preprocessing.

#### Multiple sheets detected — sheet selection dialogue
1. Inform the user that multiple sheets were detected and list all sheet names with **checkboxes** next to each.
2. The R command run in the background (shown in the live code panel) is:
   ```r
   readxl::excel_sheets("uploaded_file.xlsx")
   ```
3. If the user selects **one sheet**, pass the sheet name to the `sheet` argument of `read_excel`, import that sheet only, and proceed with standard preprocessing:
   ```r
   readxl::read_excel("uploaded_file.xlsx", sheet = "SheetName")
   ```
4. The option to **cancel and go back to single-sheet selection** is always available.

#### Multiple sheets selected — merge dialogue
If the user selects **two or more sheets**, the app runs separate `read_excel` calls for each:
```r
df1 <- readxl::read_excel("uploaded_file.xlsx", sheet = "Sheet1")
df2 <- readxl::read_excel("uploaded_file.xlsx", sheet = "Sheet2")
```

Then initiates a **merge dialogue** with the following steps:

1. **Identify a common subject-level identifier** — ask the user to specify which column is the unique subject ID (e.g. `Subject_ID`, `ID`, `PatientID`). The app can suggest candidates by looking for column names shared across all selected sheets.

2. **Quality control checks — report to user before merging:**
   - Are the ID columns present in all selected sheets?
   - How many subjects are in each sheet? (e.g. *"Sheet 1: 176 subjects, Sheet 2: 150 subjects"*)
   - How many IDs match across all sheets? (e.g. *"150 IDs are common to all selected sheets; 26 subjects in Sheet 1 have no match in Sheet 2"*)

3. **Merge options — user chooses:**
   - **Keep matched subjects only** (`inner_join` on intersection): merged dataset contains only the 150 matching subjects. Unmatched rows are dropped with the user's confirmation.
   - **Keep all subjects, fill missing with blank**: merged dataset retains all 176 subjects from the larger sheet; unmatched rows produce `NA` values for the missing columns.

4. **Confirmation before proceeding** — show a summary (e.g. *"Merged dataset will have 150 rows and X columns"* or *"Merged dataset will have 176 rows with NA values for 26 subjects in Sheet 2 columns"*) and require the user to confirm before the merged tibble is passed to TernTables preprocessing.

**Key principle — transparency throughout:** every `read_excel`, `inner_join`, or `left_join` call performed during this process must appear in the live R code panel so the user can reproduce the data preparation step outside the app.

The option to **abort merging and go back to single-sheet selection** is always available at any point in the dialogue.
