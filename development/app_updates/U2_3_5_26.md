# New features to integrate into app 3/5/26

## Structured Footnotes: `abbreviation_footnote`, `variable_footnote`, `index_style` (v1.6.3.9008)

**What changed in the package:**

- Three new parameters added to `ternD()`, `ternG()`, `word_export()`, and `ternB()`:

  #### `abbreviation_footnote`
  - Character string (or character vector, which is collapsed with spaces)
  - Printed as the **first line** of the Word table footer, before any variable definitions or legacy footnote text
  - Intended for abbreviation keys, e.g. `"SD: standard deviation; IQR: interquartile range; BMI: body mass index"`
  - Passed through `ternB_meta` automatically — each table in a multi-table `ternB()` output carries its own abbreviation footnote

  #### `variable_footnote`
  - **Named character vector** — the name is the display variable name exactly as it appears in column 1 of the table (case-insensitive match applied as fallback); the value is the definition/annotation text
  - Example: `c("BMI" = "Measured at enrolment", "Age" = "Years at time of diagnosis")`
  - The package **auto-assigns symbols** in sequence to each named variable in the order they are provided:
    - Default (`index_style = "symbols"`): `*`, `†`, `‡`, `§`, `¶`, `∥`, then doubled `**`, `††`, `‡‡` …
    - Alphabet (`index_style = "alphabet"`): Unicode superscript letters `ᵃ`, `ᵇ`, `ᶜ` …
  - Each symbol is **appended to the variable name in column 1** of the flextable at the row where that variable appears
    - `*` is appended as plain text (it naturally reads as superscript-like in most fonts)
    - `†`, `‡`, `§`, etc. are applied as **true Word superscripts** using `flextable::compose()` + `fp_text(vertical.align = "superscript")`
    - Alphabet symbols are always plain-text append (Unicode glyphs are inherently raised)
  - In the **footnote body** the symbol appears as plain text (standard journal convention): e.g. `"† Measured at enrolment"`
  - Passed through `ternB_meta` — each table in a `ternB()` multi-table output carries its own `variable_footnote` independently

  #### `index_style`
  - Character; `"symbols"` (default) or `"alphabet"`
  - Controls the auto-assigned symbol sequence for `variable_footnote` entries
  - Has no effect when `variable_footnote = NULL`

- **Footnote assembly order in the Word footer** (when any are present):
  1. `abbreviation_footnote` text (if provided)
  2. Variable definition lines — one per entry: `"* Definition text"`, `"† Definition text"`, …
  3. Legacy `table_footnote` lines (existing free-text footnote — unchanged, still supported)
  
- The footer block (double border, 6 pt Arial italic) is only added when at least one of the three is non-empty — no empty footer is ever written

- The old `table_footnote` parameter is **fully backwards-compatible** and still works exactly as before; it just now appends after the new structured lines rather than being the only footnote content

**App changes needed:**

The current footnote UI is a **single open text input** that maps to `table_footnote`. This should be **replaced by a three-section structured footnote panel**:

---

### Section 1 — Abbreviation Footnote

- **UI element:** Single text area / text input, labelled *"Abbreviation footnote"*
- **Helper text:** *"Lists abbreviations used in the table. Printed first in the footer, e.g.: SD: standard deviation; IQR: interquartile range"*
- Maps directly to the `abbreviation_footnote` argument as a character string
- Default: empty (no abbreviation line printed)

---

### Section 2 — Variable Footnote Builder

- **UI element:** A dynamic **add/remove row builder**, not a free-text box
- Each row has:
  - A **dropdown / select** populated with the variable names currently selected for the table — labelled *"Variable"*
    - The options should reflect the display names as they appear in column 1 (i.e. after `smart_rename` processing if applicable)
    - Only variables actually in the current table should be selectable (not the full dataset column list)
  - A **text input** for the definition/annotation — labelled *"Footnote text"*
  - A **remove (×) button** to delete that row
- An **"+ Add variable footnote"** button appends a new empty row
- The order the rows appear in the builder determines the symbol assignment order (`*` goes to the first row, `†` to the second, etc.) — UI should make this clear with a small note or the assigned symbol should preview live next to each row
- Maps to `variable_footnote` as a named character vector constructed from the builder rows:
  ```r
  # Example from 2 builder rows:
  variable_footnote = c(
    "BMI"      = "Measured at enrolment",
    "Age"      = "Years at time of diagnosis"
  )
  ```
- Default: empty builder (no variable symbols)

- **Index style selector** (shown alongside the builder or as a sub-option beneath it):
  - Label: *"Symbol style"*
  - Radio or compact select with two options:
    - `"Symbols"` → `index_style = "symbols"` *(recommended default)*
    - `"Alphabet (ᵃ ᵇ ᶜ)"` → `index_style = "alphabet"`
  - Default: `"Symbols"`
  - Only needs to be shown/active when at least one variable footnote row exists

---

### Section 3 — General Footnote (legacy `table_footnote`)

- **UI element:** The existing free-text input, **relabelled** *"General footnote"* (previously just "Footnote" or similar)
- **Helper text:** *"Any additional free text printed last in the footer, below abbreviations and variable definitions."*
- Behaviour unchanged — maps to `table_footnote` exactly as before
- Default: empty

---

### R call construction

When building the `ternD()` / `ternG()` call from app state, include all three args explicitly:

```r
ternG(
  ...,
  abbreviation_footnote = "SD: standard deviation; IQR: interquartile range",
  variable_footnote = c(
    "BMI"  = "Measured at enrolment",
    "Age"  = "Years at time of diagnosis"
  ),
  index_style = "symbols",          # or "alphabet"
  table_footnote = "Free text here" # legacy, still supported
)
```

- When `abbreviation_footnote` is empty/NULL, omit it from the generated script
- When the `variable_footnote` builder has no rows, omit `variable_footnote` and `index_style` from the script
- When `table_footnote` is empty, omit it (existing behaviour)

---

### Downloaded script reproducibility

- The downloaded `.R` script must include all non-NULL footnote arguments explicitly so that the output is fully reproducible outside the app
- The `variable_footnote` named vector should be written out with named elements, not positionally, for readability:
  ```r
  variable_footnote = c(
    "BMI" = "Measured at enrolment",
    "Age" = "Years at time of diagnosis"
  )
  ```

---

### `ternB()` / multi-table context

- Each table in a `ternB()` call is produced by a prior `ternD()` or `ternG()` call — the footnote parameters are already stored in `ternB_meta` per table
- No special handling is needed at the `ternB()` call site in the app; footnotes are carried automatically
- However, if the app ever allows editing footnotes at the `ternB()` stage (e.g. a "combined table footnote"), that would map to `table_footnote` on the `ternB()` call directly — the structured footnote params are intentionally set per individual table at the `ternD/ternG` stage
